Technical ML / MLOps Retail AI

"Wegmans wants to use AI/ML — demand forecasting, supply chain optimization, personalized promotions. How would you architect Databricks to support ML workloads alongside analytics? And how do you think about MLOps governance?"

Wegmans Solutions Architect Prep  ·  Steve Lysik  ·  ~4 min answer
Pocket Answer 30–45 sec
"The key insight is that ML doesn't need a separate platform — it's a consumer of the same lakehouse we already built for analytics. Gold-layer tables become Feature Store tables in Unity Catalog — daily sales by SKU by store, inventory velocity, promotional lift. Data scientists train models using those governed feature tables in Databricks notebooks, track experiments in MLflow, and register production-ready models in the Unity Catalog Model Registry. That gives us full lineage — which data trained which model, which model serves which predictions. For serving, Model Serving endpoints handle real-time inference — a demand forecast API that store replenishment systems call directly. MLOps governance mirrors data governance: models have the same classification tiers, A/B testing via champion-challenger in the registry, and automated monitoring for drift detection. I built similar patterns at Microsoft for Fortune 500 accounts, and my Agentic AI certification covers exactly this lifecycle."

Scripted Answer — Four Parts
Part 1 — One Platform, Not Two (30s)
"The most important architectural decision is: ML is not a separate platform. It's a consumer of the same lakehouse that serves analytics. The Gold-layer tables we built for BI — daily sales, inventory positions, loyalty segments — those become the Feature Store. A data scientist building a demand forecast model uses the same governed dataset that a merchandising analyst uses in Power BI.

This eliminates the biggest anti-pattern I've seen in enterprise ML: data scientists copying data into their own environment, creating ungoverned shadow copies, and training models on stale or inconsistent data. One source of truth, two consumption patterns."
Part 2 — Feature Engineering + Training (90s)
"For Wegmans specifically, let me walk through a demand forecasting use case.

Feature engineering happens in the lakehouse. I'd create Feature Store tables in Unity Catalog — wegmans_ml.features.daily_sku_store_sales — that combine SAP sales data with weather data, promotional calendars, local events, and holiday schedules. These are managed as point-in-time correct features, which is critical — you can't leak future data into a forecast model. Databricks Feature Store handles this natively with time-series–aware lookups.

Training uses MLflow on Databricks. Data scientists run experiments in notebooks, MLflow automatically logs hyperparameters, metrics, and the exact dataset version. Every experiment is reproducible — you can answer 'which data trained this model?' years later because Unity Catalog tracks the full lineage: data → features → experiment → model.

For the compute side, training jobs use dedicated ML clusters with GPU support, separate from the SQL Warehouses serving analytics. This is important — a long-running training job shouldn't impact dashboard refresh times. Auto-scaling clusters spin down when not training."
Your Experience Bridge

"At Microsoft, I designed ML-ready data platforms for ExxonMobil and FedEx — feature engineering on top of the same medallion architecture. The pattern is identical: Gold tables serve both BI and ML. And my Agentic AI certification covers exactly this model lifecycle, including the governance layer that connects to MCP Servers."

Part 3 — Model Registry + Serving (60s)
"Once a model performs well in experiments, it gets registered in the Unity Catalog Model Registry. This is the central point of control.

Models go through stages: None → Staging → Champion. A new demand forecast model starts as a Challenger — it runs in parallel against the current Champion model, and we compare predictions against actuals. Only when the Challenger consistently outperforms does it get promoted. This is automated A/B testing for models.

For serving, Databricks has two paths. Batch inference — a nightly job scores the demand forecast for every SKU at every store for the next 7 days, writes predictions back to a Gold table that store replenishment systems consume. Real-time serving — Model Serving endpoints expose a REST API. A smart cart app or a promotional engine can call the endpoint with a customer's loyalty segment and get personalized recommendations in milliseconds.

Both paths are governed — the model's service principal has specific Unity Catalog grants, every inference is logged."
Part 4 — MLOps Governance: Why It Matters for Retail (60s)
"MLOps governance is the part most organizations skip, and it's the part that breaks things in production. Three pillars.

Lineage and reproducibility — Unity Catalog tracks which data version trained which model. If a demand forecast starts performing poorly, I can trace back to the exact feature set, training data, and hyperparameters. Was there a data quality issue in the SAP feed? Did a feature definition change? Lineage answers that in minutes instead of days.

Drift monitoring — Lakehouse Monitoring watches for both data drift (input distributions shifting — say, a new product category launches) and prediction drift (model accuracy degrading over time). When drift exceeds a threshold, it triggers an automated retraining pipeline. Grocery retail is seasonal and volatile — a model trained on summer data won't predict holiday demand accurately. Continuous monitoring catches this before it impacts replenishment decisions.

Access control on models — models get the same governance tiers as data. A demand forecast model that uses only Internal-tier aggregated data can be deployed broadly. A personalization model that uses Confidential-tier loyalty data has restricted deployment and stricter audit requirements. The governance framework I described for data extends naturally to models."
Your Experience Bridge

"At Hopewell, I'm building NL2SQL prototypes with vector indexes — which is the AI consumption layer. The pattern is the same: governed data, governed models, governed access. MCP Servers in Wegmans' JD are the interface between AI agents and the governed platform. The ML platform and the AI agent platform converge on Unity Catalog as the single governance layer."

Likely Follow-Ups
"How do you handle the cold start problem? We don't have historical ML features yet." ▼
"You actually have more features than you think — they're just in your analytics tables, not labeled as features yet. Daily sales by SKU by store? That's your primary demand signal. SAP inventory levels? That's your stock feature. Weather data you're already ingesting for operations? That's your external signal.

The cold start isn't about collecting data — it's about restructuring existing Gold tables as point-in-time correct features. I'd start with the demand forecasting use case because you have the most historical data there (years of POS transactions), and it has the clearest ROI — reducing perishable waste directly impacts margins.

You can have a production demand forecast model within 8-12 weeks of starting, because the data is already there. The Feature Store just formalizes the pipeline."
"How does this relate to the MCP Servers mentioned in the job description?" ▼
"MCP Servers are the interface layer between AI agents and the governed platform. Think of it this way: a Databricks SQL-managed MCP Server exposes your Gold catalog and model predictions to AI agents — a conversational agent that a store manager asks 'what should I reorder this week?' queries both the demand forecast predictions and the current inventory table, all governed through Unity Catalog.

The ML platform produces the predictions. MCP Servers make those predictions consumable by AI agents. Unity Catalog governs both. It's three layers working together: data platform → ML platform → AI agent layer, all on the same Databricks infrastructure."
"How do you ensure the personalization models don't create privacy issues with loyalty data?" ▼
"This connects directly to the data governance framework. Personalization models train on Confidential-tier loyalty data, so they get Confidential-tier governance.

Practically: the Feature Store tables for loyalty features use hashed customer IDs — the model never sees names or emails, just pseudonymous behavioral signals. Column masking ensures that even the data scientists building the model can't join back to PII. The model outputs (recommended products, promotional offers) are predictions, not personal data, so they can flow to the promotional engine at Internal tier.

For compliance: Unity Catalog audit logs track every access to loyalty feature tables. If a customer exercises data rights, you can trace exactly which models used their data through lineage. Privacy by design, not privacy as an afterthought."
"What's your approach to compute cost management for ML workloads?" ▼
"ML compute is the most expensive part if not managed carefully. Three strategies.

Separate compute pools — ML training clusters are isolated from SQL Warehouses. Training jobs use auto-terminating clusters that spin down after idle periods. No GPU clusters running 24/7 waiting for the next experiment.

Batch over real-time — most retail ML use cases are batch. Demand forecasts run nightly, not per-request. Personalization scores refresh daily. Real-time serving is reserved for the few use cases that genuinely need millisecond latency (smart cart recommendations). This saves significant compute cost.

Feature Store as cost saver — computed features are stored, not recomputed. If five different models use 'trailing 30-day sales by SKU,' it's computed once and shared. Without a Feature Store, every data scientist computes it independently, wasting cluster hours."
