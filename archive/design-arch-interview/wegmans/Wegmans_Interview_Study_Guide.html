Part 1 — Frame the Problem (30s)
What You Say

"Before I draw anything, let me frame the problem. A grocery retailer like Wegmans has four primary data domains: transactional data from SAP — inventory, sales, finance, procurement. Real-time operational data from POS and smart carts. Digital channel data from the website and app. And external data like weather and supply chain feeds. The architecture needs to unify all of these into a single governed platform that serves both operational dashboards and advanced analytics like demand forecasting."

Part 2 — Ingestion (60s)
What You Say

"For ingestion, I'd split into two paths. Batch and CDC for SAP — using Azure Data Factory with the SAP CDC connector, which uses ODP for delta-aware extraction. Only changed records move, not full table dumps. SAP extracts run hourly for high-velocity tables, daily for master data.

For real-time, POS transactions and smart cart events flow through Azure Event Hubs into Databricks Structured Streaming. Sub-minute latency for operational use cases like real-time inventory position by store.

Both paths land in the same Delta tables — Delta Lake's ACID transactions handle the merge cleanly. This is something I designed at Microsoft for FedEx, where we had the same pattern of CDC batch plus real-time event streams."

Part 3 — Medallion Layers (90s)
What You Say

"Bronze is raw. Everything lands exactly as it arrived — SAP ODP deltas, POS JSON, clickstream. Schema-on-read, append-only. Full history for replay and audit. This is your insurance policy.

Silver is one version of the truth. Deduplication, null handling, cross-source joins — matching SAP sales records with POS transactions. Conformed dimensions: product, store, supplier, customer. SCD Type 2 for slowly changing dimensions. Delta Live Tables with quality expectations at every transition — null rates below 1%, referential integrity checks.

Gold is business-ready. Daily sales by store by category by SKU. Real-time inventory position. Supplier performance scorecards. Margin analysis. Z-ordered on common query patterns — store_id and date for most retail analytics. Feature tables for ML models live here too.

All three layers managed by DLT as declarative pipelines. You define what the data should look like, not how to get there."

Part 4 — Governance (60s)
What You Say

"Governance spans all layers through Unity Catalog. Three-level namespace — wegmans_retail.sales.daily_transactions. Column-level lineage traces any Gold metric back to Bronze. Row and column masking for PII. Audit logging for compliance.

Unity Catalog isn't just access control — it's the foundation for AI governance too. ML model lineage tracks which data trained which model. And with Databricks' SQL-managed MCP servers, AI agents query through Unity Catalog — governed access, not raw table access."

Part 5 — Consumption + Close (60s)
What You Say

"For consumption: store managers get Power BI via Databricks SQL Warehouses. Data team gets SQL for ad-hoc analysis. Data scientists access feature tables for demand forecasting, waste prediction, price optimization.

And the return path: Databricks pushes enriched data — forecasts, supplier scores — back to SAP BDC via Delta Sharing. The data doesn't flow one direction — it's a round trip.

This is the same medallion pattern I've deployed across Fortune 500 at Microsoft — FedEx, UPS, ExxonMobil. The data sources differ, but the architecture principles are identical."

"How do you handle schema changes from SAP?"
Delta Lake schema evolution in Bronze (mergeSchema). DLT quality gates in Silver catch breaking changes before propagation. Alert on schema drift, don't silently fail.
"What about data freshness SLAs?"
Different layers, different SLAs. Bronze: minutes (streaming) or <1 hour (batch). Silver: 15–30 min. Gold: depends on use case — real-time inventory near-instant, forecasting refreshes daily. DLT monitors freshness against SLAs.
"How do you handle late-arriving data?"
Streaming watermarks define how late you accept events. For batch, Delta merge (UPSERT) handles late arrivals. Bronze is append-only — nothing lost. Silver merge reconciles.
"How would you size compute?"
Serverless SQL Warehouses for BI — auto-scale to zero. Jobs clusters with auto-scaling for DLT. Separate compute for ML training. Start small, monitor, scale on actual usage.
TECHNICAL
How would you migrate from a legacy SQL Server BI stack to a cloud lakehouse?
▾
Target: 3–4 minutes
What You Say

"This is actually the core of what I did at IBM, just with different source systems. At IBM, I migrated 50TB+ from legacy platforms with 3x performance improvement. The approach was the strangler fig pattern — three phases.

Phase 1: Parallel run. Replicate existing SSRS reports from the new cloud platform. Don't turn off anything — just prove parity. This builds trust with the business users who depend on those reports daily.

Phase 2: Migrate the pipelines. Move ETL from SSIS to Databricks Delta Live Tables. SSRS reports migrate to Power BI connected to Databricks SQL Warehouses. This is where most of the performance gains come from — DLT optimizes the execution, and Databricks SQL is significantly faster than SSRS on SSAS cubes.

Phase 3: Sunset legacy. Only after consumers have migrated and you've run automated data quality checks comparing legacy vs new outputs for 3–6 months.

The critical lesson from my IBM experience: you can't tell store managers their reports are temporarily unavailable. Dual-run is non-negotiable. The business runs 24/7, so the migration has to be invisible to end users."

Your Proof Points
  • 50TB+ migration at IBM with zero production outages
  • 3x performance improvement on the target platform
  • 100% POC win rate for subsequent expansion
  • "Strangler fig pattern" — use the name, it shows architectural vocabulary

What You Say

"MCP — Model Context Protocol — enables AI agents to share context and query governed data through a standardized interface. In the Wegmans context, think about multiple AI agents working together: a demand forecasting agent, a pricing optimization agent, and a supplier scoring agent — all accessing the same governed Databricks lakehouse via MCP servers fronting Unity Catalog.

Databricks released SQL-managed MCP servers in late 2025, so this is production-ready infrastructure, not theoretical. The MCP server becomes the governed API layer between AI agents and your data — agents get access to the data they're authorized for, with full audit logging.

At Hopewell, I built NL2SQL prototypes using vector indexes for semantic data discovery — MCP servers are the next evolution of that same pattern, giving AI agents governed access to enterprise data rather than direct table queries."
