Wegmans Lakehouse Architecture

Interview Reference — "Walk us through how you'd design a data lakehouse architecture on Databricks for a large retail organization."

Pocket Answer 30–45 sec
"I'd build a Delta Lakehouse on Databricks with a medallion architecture — Bronze for raw ingestion, Silver for cleansed and conformed, Gold for business-ready analytics. Ingestion splits two ways: batch CDC from SAP via Azure Data Factory's SAP connector for inventory, sales, and finance data, and real-time streaming from POS and smart carts through Event Hubs into Structured Streaming. Unity Catalog governs everything — three-level namespace, column-level lineage, row masking for PII. Delta Live Tables manage pipeline orchestration with data quality expectations at every layer transition. Consumption serves Power BI for store managers, SQL for analysts, and feature tables for ML models like demand forecasting. And critically, it's a round trip — enriched data flows back to SAP via Delta Sharing. I built this exact pattern at Microsoft for FedEx and ExxonMobil."

Your Answer — Scripted with Transitions

Practice this as a 5–7 minute walkthrough. Use the diagram above as your whiteboard reference. Each part is a natural pause point where they might ask follow-up questions.

Part 1 — Frame the Problem (30 sec)
What You Say

"Before I draw anything, let me frame the problem. A grocery retailer like Wegmans has four primary data domains: transactional data from SAP — inventory, sales, finance, procurement — that's your system of record. Real-time operational data from POS systems and smart carts. Digital channel data from the website and app. And external data like weather and supply chain feeds. The architecture needs to unify all of these into a single governed platform that serves both operational dashboards and advanced analytics like demand forecasting. That's what the lakehouse gives us."

Why This Opening Works
  • Shows you think about business context before technology
  • Demonstrates retail domain knowledge (the four data domains)
  • Names SAP specifically — signals you understand their stack
  • Sets up the architecture rationale: unification + governance + analytics
Part 2 — Ingestion: Batch + Real-Time (60 sec)
What You Say

"For ingestion, I'd split into two paths. Batch and CDC for SAP data — using Azure Data Factory with the SAP CDC connector, which uses ODP for delta-aware extraction. This means we're only moving changed records, not full table dumps. I'd schedule SAP extracts hourly for high-velocity tables like sales and inventory, daily for slower-moving data like supplier master records.

For real-time, POS transactions and smart cart events flow through Azure Event Hubs into Databricks Structured Streaming. This gives us sub-minute latency for operational use cases — like real-time inventory position by store, which is critical for omnichannel fulfillment.

Both paths land in ADLS Gen2 as Delta Lake tables. The key architectural decision here is that batch and streaming write to the same Delta tables — Delta Lake's ACID transactions handle the merge cleanly. This is something I designed at Microsoft for FedEx, where we had the same pattern of CDC batch plus real-time event streams landing in a unified lakehouse."

Why This Section Works
  • Addresses both batch AND real-time — directly answers what they asked
  • SAP CDC + ODP shows you know the extraction pattern for their stack
  • The "same Delta tables" point is architecturally important — no separate real-time vs batch stores
  • FedEx reference = proof you've built this at comparable scale
Part 3 — Medallion Architecture (90 sec)
What You Say

"I'd structure the lakehouse in three layers — the medallion architecture. Bronze is raw. Everything lands exactly as it arrived — raw SAP ODP deltas, POS JSON, clickstream events. Schema-on-read, append-only. We keep full history here for replay and audit. This is your insurance policy.

Silver is where we create one version of the truth. Deduplication, null handling, cross-source joins — matching SAP sales records with POS transactions, for example. We build conformed dimensions here: product, store, supplier, customer. I'd use SCD Type 2 for slowly changing dimensions like product attributes or store details. This is also where Delta Live Tables earns its keep — I'd set data quality expectations at every transition. For example, null rates below 1% on key fields, referential integrity between fact and dimension tables.

Gold is business-ready. Pre-aggregated tables purpose-built for specific use cases: daily sales by store by category by SKU for demand forecasting, real-time inventory position for operations dashboards, supplier performance scorecards, margin analysis by product and store. I'd Z-order Gold tables on the most common query patterns — probably store_id and date for most retail analytics. And this is where feature tables live for ML models — structured so data scientists can pull training data efficiently.

All three layers are managed by Delta Live Tables as declarative pipelines. The advantage is that DLT handles orchestration, retries, and data quality monitoring — you define what the data should look like, not how to get there. That's a significant operational simplification at scale."

Why This Section Works
  • Each layer has a clear purpose AND retail-specific examples — not generic
  • DLT quality expectations show you think about data reliability, not just data movement
  • Z-ordering is a Databricks-specific optimization — shows platform depth
  • Feature tables for ML ties the architecture to their analytics ambitions
  • The "insurance policy" framing for Bronze resonates with risk-conscious enterprises
Part 4 — Governance (60 sec)
What You Say

"Governance spans all three layers through Unity Catalog. Three-level namespace — catalog, schema, table — so we can organize by domain: wegmans_retail.sales.daily_transactions, for example. Column-level lineage means we can trace any Gold metric back through Silver to its Bronze source. Row and column masking for PII — customer data, employee data. Audit logging for compliance.

One thing I'd call out specifically: Unity Catalog isn't just access control. It's the foundation for AI governance too. When we build ML models on demand forecasting or pricing, the lineage tracks which data trained which model version. And with Databricks' SQL-managed MCP servers, AI agents query through Unity Catalog — they get governed access, not raw table access. That ties into the MCP architecture from the job description."

Why This Section Works
  • Concrete namespace example — not abstract
  • Connects governance to AI/ML — forward-looking, not just compliance
  • MCP callback to the JD shows you read it carefully
  • PII masking is relevant (Wegmans collects customer data via loyalty programs and biometrics)
Part 5 — Consumption + Feedback Loop (60 sec)
What You Say

"For consumption, the lakehouse serves multiple personas. Store managers get operational dashboards via Power BI connected to Databricks SQL Warehouses — inventory levels, sales trends, staffing indicators. The data team gets SQL-based access for ad-hoc analysis. Data scientists access feature tables for ML — demand forecasting, waste prediction, price optimization.

And here's the piece that closes the loop: the return path to SAP. Databricks pushes enriched data — demand forecasts, supplier scores, ML predictions — back to SAP Business Data Cloud via Delta Sharing. So the forecasting model we train in Databricks directly informs the replenishment logic running in SAP. The data doesn't just flow in one direction — it's a round trip."

Why This Section Works
  • Multiple consumption patterns show you think about all stakeholders
  • The bidirectional SAP flow is a differentiating point — most candidates only think about ingestion
  • Connecting ML predictions to SAP replenishment shows you understand the business impact
  • Strong close: "round trip" is memorable and architecturally accurate
Close — Tie It Back (15 sec)
What You Say

"This is the same medallion pattern I've designed and deployed across Fortune 500 accounts at Microsoft — FedEx, UPS, ExxonMobil. The data sources differ, but the architectural principles are identical. The key advantage of this approach for Wegmans is that you get a single platform for everything from operational dashboards to demand forecasting to AI agents — governed consistently, scaled efficiently."

Likely Follow-Up Questions

"How would you handle schema changes from SAP?"

Delta Lake schema evolution in Bronze (mergeSchema option), with DLT quality gates in Silver to catch breaking changes before they propagate. Alert on schema drift, don't silently fail.

"What about data freshness SLAs?"

Different layers have different SLAs. Bronze: within minutes of source change (streaming) or within 1 hour (batch CDC). Silver: typically 15–30 minutes behind Bronze. Gold: depends on use case — real-time inventory is near-instant, demand forecasting aggregates refresh daily. DLT pipeline monitoring tracks freshness against SLAs.

"How do you handle late-arriving data?"

Structured Streaming watermarks for the real-time path — define how late you'll accept events before dropping them. For batch, Delta Lake's merge (UPSERT) handles late arrivals cleanly. Bronze is append-only so nothing is ever lost; Silver merge logic reconciles.

"How would you size the compute?"

Serverless SQL Warehouses for BI/SQL consumption — auto-scale to zero when not in use, no cluster management. Jobs clusters for DLT pipelines with auto-scaling. Separate compute for ML training. Unity Catalog governs who can use which compute pools. Start small, monitor, scale based on actual usage — not theoretical projections.
