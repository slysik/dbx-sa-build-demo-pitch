1

Primary Keys & Uniqueness

Constraints that inform vs. constraints that enforce
Teradata

PK = enforced. Duplicate INSERT = hard error. The engine guarantees uniqueness.

INSERT INTO customer VALUES (1, 'Alice'); INSERT INTO customer VALUES (1, 'Bob'); -- Result: ERROR. Constraint enforced.
Databricks Fix

PK is informational only. Enforce uniqueness via MERGE from staging tables in your pipeline.

MERGE INTO customer t USING staging_customer s ON t.customer_id = s.customer_id WHEN MATCHED THEN UPDATE SET * WHEN NOT MATCHED THEN INSERT *
Modern: Spark Declarative Pipelines (AUTO CDC)

4 lines replace 50 lines of manual MERGE. Keys enforce identity, sequence handles ordering, SCD Type 2 preserves full audit history.

AUTO CDC INTO gold.dim_customer FROM STREAM silver.customers_clean KEYS (customer_id) SEQUENCE BY updated_timestamp STORED AS SCD TYPE 2;

Duplicate customer records = "customer count" becomes an opinion, not a fact. Regulatory reporting requires single source of truth.

2

Duplicate Rejection (SET Tables)

Automatic dedup vs. explicit dedup
Teradata

SET tables automatically reject full-row duplicates on insert. No extra logic needed.

CREATE SET TABLE orders ( order_id INTEGER, amount DECIMAL(10,2) ); -- Duplicate rows silently rejected
Databricks Fix

Use SELECT DISTINCT for batch or dropDuplicates() for streaming to explicitly remove duplicates.

-- Batch CREATE OR REPLACE TABLE orders_clean AS SELECT DISTINCT * FROM staging_orders; -- Streaming df.dropDuplicates(["order_id"])
Modern: AUTO CDC handles dedup automatically

KEYS clause defines what makes a record unique. SEQUENCE BY resolves out-of-order arrivals. The platform handles dedup — you declare intent.

AUTO CDC INTO gold.fact_transactions FROM STREAM silver.transactions_clean KEYS (transaction_id) SEQUENCE BY event_timestamp;

Duplicate transactions in a banking ledger inflate balances. Duplicates accumulate quietly until reconciliation exposes the gap.

3

Collation & Case Sensitivity

The quiet way to break trust
Teradata

Case-insensitive by default. WHERE name = 'alice' matches Alice, ALICE, alice.

SELECT * FROM customers WHERE name = 'alice'; -- Matches: Alice, ALICE, alice
Databricks Fix

Define a collation strategy. Don't sprinkle LOWER() everywhere — document comparison rules centrally.

-- Bad: ad-hoc workaround ON LOWER(a.name) = LOWER(b.name) -- Good: collation strategy -- Normalize at Silver layer ingestion -- Document rules in data contract
Modern: COLLATE — one line, solves the whole problem

Declare case-insensitive collation at the column level. No LOWER() wrappers, no Silver-layer normalization, no drift across teams. Schema-level intent.

CREATE TABLE customers ( customer_id INT, name STRING COLLATE UTF8_LCASE, email STRING COLLATE UTF8_LCASE ); -- Now this just works: SELECT * FROM customers WHERE name = 'alice'; -- Matches: Alice, ALICE, alice

JOINs silently drop rows post-migration. Distinct counts drift. Analysts add workarounds. Dashboards look "randomly" wrong.

4

DECIMAL Precision Drift

"It loaded" is not the same as "it's right"
Teradata

Strict typing with DECIMAL(38,10). Precision preserved through aggregations. What you put in is what you get out.

-- Teradata fact table amount DECIMAL(38,10) SELECT SUM(amount) FROM fact_sales; -- Precise to 10 decimal places
Databricks Fix

Explicit CAST on aggregations + reconciliation tables to catch drift. Test pathological rows (max precision, negatives, nulls).

SELECT CAST(SUM(amount) AS DECIMAL(38,10)) FROM fact_sales; -- Reconciliation check SELECT old_sum, new_sum, old_sum - new_sum AS delta FROM reconciliation_table;
Modern: Double CAST — input AND output

Cast the input before aggregation AND the output after. Prevents Spark from silently promoting to DOUBLE mid-calculation. Matches Teradata to the penny.

-- Double CAST: prevent mid-calc promotion SELECT CAST( SUM(CAST(amount AS DECIMAL(38,10))) AS DECIMAL(38,10) ) FROM fact_sales;

A 0.01% rounding drift in transaction totals = regulatory audit finding. In FinServ, "close enough" doesn't exist.

5

ACID Transactions

All-or-nothing vs. one-at-a-time
Teradata

Multi-statement transactions are all-or-nothing. If step 3 of 4 fails, everything rolls back. No partial state.

BEGIN TRANSACTION; UPDATE dimension ... INSERT INTO facts ... INSERT INTO audit_log ... UPDATE completion_flag ... COMMIT; -- All succeed or all fail
Databricks Fix

Design for atomicity explicitly: stage → validate → publish via REPLACE TABLE or Delta version swap. Make pipelines idempotent.

-- 1. Stage CREATE TABLE fact_staging AS SELECT ... -- 2. Validate SELECT COUNT(*) FROM fact_staging; -- 3. Publish atomically REPLACE TABLE fact_production AS SELECT * FROM fact_staging;

Half-published facts in a trading system or risk model = incorrect exposure calculations. Partial state is unacceptable in regulated environments.

6

Time Zones

Decide early or pay forever
Teradata

Native TIMESTAMP WITH TIME ZONE support. Time zone semantics are explicit and enforced at the type level.

INTERVAL '01:30:00' HOUR TO SECOND TIMESTAMP WITH TIME ZONE -- Zone-aware by design
Databricks Fix

Store UTC instant + original timezone string. Convert at display time. Decide what timestamps mean before you store them.

-- Store event instant event_ts_utc TIMESTAMP -- Store original zone event_tz STRING -- e.g., "America/New_York" -- Display conversion SELECT from_utc_timestamp( event_ts_utc, event_tz )

Global banks operate across time zones. DST shifts cause audit drift and inconsistent trade timestamps. A semantic contract, not a technical detail.

7

SQL Differences

The death of a thousand paper cuts
Teradata

Volatile tables, macros, triggers, UPDATE...FROM...SET, stored procedures. Decades of procedural patterns baked in.

-- Volatile tables CREATE VOLATILE TABLE temp AS (...); -- Update-with-join UPDATE target t FROM source s SET t.value = s.value WHERE t.id = s.id;
Databricks Fix

Systematic rewrite strategy: TEMP VIEWs, MERGE INTO, Databricks SQL scripting or dbt. Decide the approach once, not per-query.

-- Volatile -> Temp View CREATE OR REPLACE TEMP VIEW temp AS (...); -- Update-join -> MERGE MERGE INTO target t USING source s ON t.id = s.id WHEN MATCHED THEN UPDATE SET value = s.value

Inconsistent procedural logic patterns across teams = maintenance costs explode. FinServ auditors expect documented, repeatable processes.
