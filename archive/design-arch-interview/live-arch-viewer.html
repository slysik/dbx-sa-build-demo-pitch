🏛️ Key Architecture Decisions
Decision Choice Rationale Trade-off
**Migration strategy** Phased domain-by-domain Reduces blast radius; Finance first (highest ROI), then Risk, then Ops; never all-or-nothing Longer coexistence period; dual-write complexity during transition
**Coexistence pattern** Lakehouse Federation + parallel run Downstream apps keep reading EDW while Gold layer is validated; cutover is a connection-string change Federation adds query latency; must maintain EDW until final domain cutover
**Netezza migration** Lakebridge Analyzer → transpiler Analyzer classifies the full SQL estate first — only migrate actively-used objects; 80% auto-transpile ~20% stored procs need manual rework; BTEQ scripts need human review
**Ingestion — EDW snapshots** Lakebridge CDC + Auto Loader Full snapshot for history load; incremental CDC from OLTP for ongoing; mainframe VSAM → flat file → Auto Loader Mainframe CDC direct requires MQ or z/OS tap — flat file extract is safer first step
**Silver modeling** 3NF Integration Layer EDW parity — existing SQL and semantic layers work against Silver with minimal changes; SCD Type 2 preserves history Not as auditable as Data Vault for multi-source reconciliation — acceptable for single-EDW migration
**Gold modeling** Kimball Star + Materialized Views Matches existing mart structure; BI tools repoint without semantic layer rebuild; MVs replace EDW aggregation tables Snowflake schema variant for complex dims; revisit if OBT pattern proves faster for ad-hoc
**Dedicated SQL Warehouses** One per domain (Finance · Risk · Ops) Mirrors EDW resource group isolation; SLA guarantees per team; no workload interference at month-end Higher cost than shared warehouse — justified by regulatory SLA requirements
**Downstream app cutover** JDBC/ODBC repoint (connection string only) Apps connect to DBSQL instead of Netezza with zero code change; validate with parallel run Requires Gold layer to be 100% query-compatible — Lakebridge validates SQL parity before cutover
🔍 Open Questions
  • ❓ **Cloud provider** — Azure, AWS, or GCP? Determines Federation connector and networking path
  • ❓ **Netezza estate size** — TB of data, number of tables, active vs. dormant object count?
  • ❓ **How many downstream apps** read directly from the EDW today — and do they use JDBC or stored proc calls?
  • ❓ **Report SLAs** — which are the top 5 reports by business criticality? These are the migration sequencing anchors
  • ❓ **Mainframe extract mechanism** — scheduled VSAM dumps, MQ feeds, or IBM CDC (IIDR)?
  • ❓ **Coexistence period** — how long is acceptable? 3 months? 12 months? Drives dual-run cost budget
  • ❓ **Semantic layer tool** — is there a Business Objects / Cognos / MicroStrategy layer sitting above the EDW?
  • ❓ **Compliance scope** — SOX only, or Basel IV capital models also running on Netezza?
  • ❓ **Migration trigger** — contract renewal, hardware EOL, or cost mandate from CFO?
