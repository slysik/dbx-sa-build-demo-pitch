## Detailed Findings

### 🔴 CRITICAL GAP 1: SQL-First Not Enforced

**Playbook says (§4,7,8,13):** "SQL-first. Python allowed ONLY to generate synthetic datasets."

**Current state:** ALL four notebooks are 100% PySpark:
- `01_Bronze` — PySpark streaming with `.withColumn()` chains
- `02_Silver` — PySpark `foreachBatch` with DataFrame API
- `03_Gold` — PySpark `.groupBy().agg()` 
- `04_MLflow` — PySpark + sklearn (appropriate for ML)

**Interview pipeline agent prompt** correctly says: "Python for data generation ONLY, SQL for ALL transforms" — ✅ aligned with playbook.

**Impact:** The agent prompt is correct, but existing notebooks don't follow it. Any NEW code generated by the pipeline will be SQL-first (good), but existing demos are all PySpark.

**Recommendation:** 
- Existing notebooks are fine as-is (they work, they demo streaming well)
- For interview coding exercises, the pipeline agents should generate SQL transforms — this is already configured correctly
- Add a note to CLAUDE.md that the existing notebooks are "PySpark demo" style vs the playbook's "SQL-first interview" style

---

### 🔴 CRITICAL GAP 2: DECIMAL vs FLOAT for Money

**Playbook says (§4.2):** `amount DECIMAL(18,2)` — "Cast amount to DECIMAL(12,2) for monetary precision — FLOAT would introduce rounding errors"

**Current notebooks:**
- Bronze: `F.round(F.rand() * F.lit(5000), 2)` → produces DoubleType
- Silver: No CAST to DECIMAL
- Gold: Aggregates on uncasted doubles

**Agent prompt** correctly says: "Use DECIMAL for monetary values, never FLOAT" ✅

**Impact:** Existing tables in `dbx_weg` catalog have DOUBLE amounts. New interview code will use DECIMAL (good).

**Recommendation:** No need to rewrite existing notebooks, but the playbook's DECIMAL pattern should be the default for all new code. Already enforced in agent prompts.

### 🔴 HIGH GAP 3: No Deterministic ROW_NUMBER Staging Before MERGE

**Playbook says (§7.1):**
```sql
CREATE OR REPLACE TEMP VIEW v_latest_per_txn AS
SELECT * FROM (
  SELECT *, row_number() OVER (PARTITION BY txn_id ORDER BY ingest_ts DESC) AS rn
  FROM finserv_bronze.payment_events
) WHERE rn = 1;

MERGE INTO finserv_silver.payment_current t
USING v_latest_per_txn s ON t.txn_id = s.txn_id ...
```

**Current notebook (02_Silver):** Uses `foreachBatch` + MERGE but no explicit ROW_NUMBER dedup staging. If duplicates arrive in the same micro-batch, the MERGE could match multiple source rows to one target → non-deterministic.

**Agent prompt** correctly says: "Deduplicate on transaction_id using ROW_NUMBER" ✅

**Recommendation:** Already covered in agent prompts. The playbook pattern should be the default for any new MERGE code. Existing notebooks could be improved but aren't broken (duplicates are rare in rate-source simulations).

### 🔴 HIGH GAP 4: No Liquid Clustering in Notebooks

**Playbook says (§5.1):**
```sql
ALTER TABLE finserv_bronze.payment_events CLUSTER BY (account_id, event_date);
OPTIMIZE finserv_bronze.payment_events;
ANALYZE TABLE finserv_bronze.payment_events COMPUTE STATISTICS;
```

**Current notebooks:** No CLUSTER BY, no OPTIMIZE, no ANALYZE anywhere. Tables are unoptimized.

**Skills coverage:**
- `databricks-spark-declarative-pipelines` mentions `cluster_by` in SDP Python API ✅
- `databricks-dbsql` mentions Liquid Clustering in data modeling ✅
- But no skill has the specific "default to CLUSTER BY (key, date) + OPTIMIZE + ANALYZE" pattern from the playbook

**Recommendation:** Add the playbook's §5 pattern as a standard post-create step in generated code. The agent prompts should include: "After creating tables, add CLUSTER BY, run OPTIMIZE, and COMPUTE STATISTICS."

### 🔴 HIGH GAP 5: No CHECK Constraints

**Current notebooks:** Zero constraints on any table.

**Agent prompt** says: "Include Delta table constraints for data quality" ✅

**Recommendation:** Already in agent instructions. The playbook's specific patterns (CHECK for amount, currency, status) should be generated by default.

### 🔴 HIGH GAP 6: No Validation Harness

**Playbook says (§11):** Run count comparisons, uniqueness checks, rule checks, and pruning-friendly queries after each stage.

**Current notebooks:** No post-pipeline validation of any kind.

**Agent prompt:** Does not explicitly mention a validation harness step.

**Recommendation:** Add to agent prompts: "Always include a validation harness as the final cells: row counts across layers, uniqueness check on business key, rule violation counts, and a sample query aligned with clustering keys."

### 🔴 HIGH GAP 7: Gold Idempotency Pattern

**Playbook says (§8):**
```sql
DELETE FROM finserv_gold.payments_agg_1d WHERE event_date >= date_sub(current_date(), 14);
INSERT INTO finserv_gold.payments_agg_1d SELECT ... WHERE event_date >= date_sub(current_date(), 14);
```

**Current notebooks (03_Gold):** `mode("overwrite")` on the entire table — rewrites all data every run. Not idempotent for a date range; just nukes and rebuilds everything.

**Recommendation:** For interview code, use the playbook's "delete window + insert window" pattern. More sophisticated and demonstrates understanding of incremental Gold rebuilds.

---

### 🟡 MEDIUM GAP 8: No AI Audit Checklist in Agent Prompts

**Playbook says (§12):** 6-point checklist:
1. Does it compile?
2. Is MERGE deterministic?
3. Is it idempotent?
4. Is it scalable?
5. Are there "why" comments?
6. Validation harness?

**Current agent prompts:** The Databricks Code Gen agent doesn't have this checklist. The Spark Explainer covers some of it (execution plan, scaling) but not all.

**Recommendation:** Embed the AI audit checklist into the Code Gen agent's prompt as a final self-check step.

### 🟡 MEDIUM GAP 9: Synthetic Data Distributions

**Playbook says (§6):** `random.random() * 5000` for amounts — but the interview-pipeline agent prompt says: "Use `random.lognormvariate()` for monetary amounts (not uniform!)"

**The playbook itself uses uniform!** `round(random.random() * 5000, 2)` in §6.

**Recommendation:** The agent prompt is actually BETTER than the playbook here. The lognormvariate instruction in the pipeline agent produces more realistic data. Keep the agent prompt as-is; consider updating the playbook.
